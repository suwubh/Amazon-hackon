# Lessons

One lesson per entry, newest on top, one-line summary first. Read this every fresh session. Don't duplicate what CLAUDE.md/architecture.md already record.

## MT4 (June 13)
- **chrome-devtools MCP won't attach if a stale Chrome from a prior session is still holding its profile** (`...\.cache\chrome-devtools-mcp\chrome-profile`) — every `navigate_page`/`new_page` errors "The browser is already running … Use --isolated". Fix: kill ONLY the MCP-owned tree, not the user's Chrome — `Get-CimInstance Win32_Process -Filter "Name='chrome.exe'" | Where CommandLine -like '*chrome-devtools-mcp*' | Stop-Process -Force`, then navigate again. (Don't blanket-kill chrome.exe.)
- **`/metrics` is cumulative over the warm Lambda instance, not per-session.** It's `BASELINE (5 items / ₹3,120 / 80%) + every GRADED/ROUTED event since the last cold start`. So routing items in testing (curl or the UI) bumps it permanently for that instance — prod read ₹3,203/83% after my test routes. Good (proves numbers are live, not hardcoded) but the inbox/metrics "this session" label drifts up across demo runs. For a clean stage number, hit a cold instance or add a reset (MT5).
- **Two new cold-start-safe wrappers, same pattern as routeSafe/healthCardSafe:** `priceCurveSafe` (price-curve 409s without a GRADED event → grade *cached* then retry; the grade is a hidden idle-asset valuation, letter is invariant, so force cached) and `routeRto` (route 409s without a SEAL_CHECKED event → re-run seal-check then route). The 1 benign 409 you'll see in the prod console on the radar lane IS priceCurveSafe's first probe — the slider still renders, so it's the retry working, not a bug.
- **Reused RouteScreen for the RTO lane via an optional `nextLabel`/`nextHint` prop** instead of a second route screen — RTO is seal→route→list (no Health Card; it's the "15-second" beat). A sealed RTO routes as grade A and `rto_relist` wins **+₹2,464** (breakdown reconciles: 2519−15−40), beating warehouse +₹1,810.
- **The sticky FooterAction's gradient is transparent at the top (where the hint sits), so on SHORT screens the hint text overlaps a real data row.** Fix that scales: added `backdrop-blur-sm` to the footer (frosted reveal — content under the CTA blurs, looks intentional on every screen) AND dropped the now-redundant hint on the shortest screen (SealLane). Long scrollable screens were always fine — content fading under the CTA is the intended look (matches the spine screens).
- **`/metrics`, `/radar`, `/price-curve`, `/seal-check`, `/diagnose-listing` all already existed from MT2** — MT4 was pure frontend, so NO backend redeploy. Only `git push` → Vercel auto-deploy. Confirm contracts with a quick curl against the Function URL before wiring (every number on screen must trace to a field).

## MT3 (June 13)
- **CORS is origin-exact: use `http://localhost:5173`, NEVER `127.0.0.1:5173`.** The Lambda `ALLOWED_ORIGINS` env = `http://localhost:5173,https://amazon-hackon.vercel.app`. A browser at `127.0.0.1:5173` is a *different* origin → every fetch dies with "Failed to fetch" (CORS, no Allow-Origin header) even though curl works fine. Drove me in a circle for one cycle. To add an origin, edit the Lambda env var — no redeploy/rebuild needed.
- **The Vercel prod URL is https://amazon-hackon.vercel.app and it auto-deploys on `git push origin main`** (was live <1 min after push; detect the new build by the `<title>` flipping from `frontend` → `Amazon Second Life`). Lambda env already had `ALLOWED_ORIGINS` + `DYNAMODB_TABLE_NAME=SecondLifePassport` set (by the human), so MT3 needed NO backend change/redeploy.
- **Baked the deployed Function URL as the api.js fallback** (`VITE_API_URL || DEPLOYED`) so the Vercel build works even if the dashboard env var is unset. `.env.local` also points at the deployed backend so `npm run dev` needs no local uvicorn. The URL is already public in api-spec.md — fine to bake.
- **Missing item photos don't break the UI:** `frontend/src/components/Thumb.jsx` has an `onError` → category-tile fallback. SL-003..008 have no images → 6 benign 404s in console, handled. Only SL-001/SL-002 ship real photos; copy them to BOTH `backend/app/seed/images/` and `frontend/public/items/SL-00X/` (frontend thumbs are static, not a backend route).
- **Frontend defends the per-instance-passport 409:** `api.routeSafe`/`api.healthCardSafe` catch a 409 (cold Lambda lost the prior GRADED/ROUTED) and re-run the prerequisites once, so the spine never dead-ends on a warm→cold swap. The natural screen order (scan→route→card) keeps one warm instance anyway.
- **DynamoDB write-through works** — after a live grade+route, `aws dynamodb query --table-name SecondLifePassport ... :i={S:SL-001}` returns GRADED + ROUTED rows (db-setup.md §4 satisfied). NOTE: `/health-card`'s documented `LISTED` event did NOT show up in DynamoDB during testing — non-blocking, but check if a listed-count ever matters.
- **Playwright MCP saves screenshots into its own sandbox** (unreachable from the Bash tool's CWD) and dumps `.playwright-mcp/` + `*.png` into the repo root — clean those before committing. For screenshots I can actually SEE inline, use the **chrome-devtools MCP** `take_screenshot` (returns the image to me) instead of playwright's (path-only).
- **Tailwind v4 `@theme` tokens worked first try** for the design system: `--color-az-navy`, `--color-sl-green`, `--font-display` in `index.css` → `bg-az-navy`, `text-sl-green`, `font-display` utilities. Fonts via Google `<link>` in index.html (Bricolage Grotesque + Plus Jakarta Sans — deliberately NOT Inter). Build is ~240KB JS / 37KB CSS, no extra deps.

## MT2 (June 13)
- **The IAM deploy blocker is RESOLVED.** `deploy.ps1` now pushes to ECR and runs `aws lambda update-function-code` successfully; the new image is live on the Function URL. Quirk: the `aws ecr get-login-password | docker login` step still prints `400 Bad Request`, but `docker push` succeeds anyway because Docker reuses cached ECR creds from a prior good login — so a 400 on the login line is NOT a deploy failure; check for `latest: digest: sha256...` and `Done. Deployed` at the end instead.
- **deploy.ps1 must run from `backend/`** (its build context is `.`). The PowerShell tool's working dir was already `backend`, so a leading `Set-Location backend` errored with `backend\backend not found` — harmless, the script still ran from the right place. If you script it, guard the cd or just run `./deploy.ps1` from backend.
- **Background shells (`run_in_background`) start in a fresh CWD, not the persisted one** — use absolute paths (`cd "C:/.../backend"`) when launching uvicorn/scripts in the background, or you get `cd: backend: No such file or directory`.
- **Windows console is cp1252 — printing the ₹ char (U+20B9) crashes with UnicodeEncodeError.** Set `PYTHONIOENCODING=utf-8` for any python that prints rupee strings (e.g. radar ping_message), or extract numeric fields instead of printing the whole JSON.
- **VRS integrity rule that MT3 depends on:** each path's `recovery` is the exact sum of its `breakdown` values (sale/credit positive, costs negative). The frontend renders the breakdown and the total must reconcile on screen. If you tune a constant, keep this invariant.
- **The api-spec /route example numbers were illustrative and didn't sum** (and assumed grade B). MT2 made them real: the example now reflects the live grade-D output for SL-001 and every figure reconciles. Recovery scales with grade — don't expect the +₹290 "hero" number unless the shoe is graded B (current placeholder cache grades it D → local +₹83).
- **Sealed RTO items skip grading** (architecture §3.2): `/route` normally 409s without a prior grade, but a verified-sealed RTO item routes as factory-new (grade A) with no grade required. So the RTO demo beat is seal-check → route (no scan).
- **In-memory passport is per-Lambda-instance.** `/route` and `/health-card` need the prior `/grade`/`/route` events to be in the SAME warm instance. Sequential demo curls reuse one instance so it works; a cold start (or parallel calls hitting different instances) loses the chain → 409. For a bulletproof stage, set `DYNAMODB_TABLE_NAME` (MT5) or always run the spine in order on one session.

## MT1 (June 13)
- **Deploy is blocked by IAM, not code.** The configured AWS CLI user `hackon-app` (account 656751413989) can invoke Bedrock (local grading works) but is **denied `ecr:GetAuthorizationToken`/ECR push and `lambda:UpdateFunctionCode`** → `deploy.ps1` builds fine, then fails at ECR login (400) and Lambda update (AccessDeniedException). A failed deploy changes nothing on the live function, so the demo path stays up — but **MT2 cannot ship to Lambda until `hackon-app` gets ECR-push + Lambda-deploy permissions, or deploy.ps1 is run with admin creds.** HUMAN TASK before MT2.
- **Nova-2 multimodal converse: image blocks go in `content` as `{"image": {"format": "jpeg", "source": {"bytes": <raw bytes>}}}`** (raw bytes, not base64), text block last. Worked first try in ca-central-1.
- **Grading consistency is solid at temperature=0.2**: hero shoe graded D/D/D across 3 live runs. The same-unit check is genuinely discriminative — placeholder current-photo (different shoe) correctly returned `same_unit.verified=false`; swap in real day-0-vs-now photos of the *same* unit before judging same-unit accuracy.
- **Git on Windows rewrites LF→CRLF** on these files (harmless warning); seed `.jpg`/`.json` commit fine and bake into the container via `COPY app` in the Dockerfile.

## Pre-build gotchas (carried from boilerplate phase — verified the hard way)
- **Bedrock throttles new accounts in us-east-1** → everything runs in ca-central-1.
- **Nova 2 Lite needs the inference-profile ID** `us.amazon.nova-2-lite-v1:0` (fallback `global.amazon.nova-2-lite-v1:0`), not a bare model ID.
- **Nova 2 is a reasoning model** → responses may contain a `reasoningContent` block before the text; extract the block that has `text`, never `content[0]`. Same applies to multimodal/JSON outputs.
- **Lambda container builds need `--provenance=false`** (handled in deploy.ps1); base image already has the RIC; x86_64, 512 MB, 30s timeout.
- **`load_dotenv()` must run before boto3 use locally**, else `NoCredentialsError`.
- **CORS lives in the FastAPI app only**; Function URL CORS stays disabled or it answers preflights itself.
- **Bedrock calls must be time-bounded** (connect 5s / read 15s, 2 retries) so a hung primary fails over to Gemini before the Lambda 30s timeout. Budget: with images, leave headroom — one live multimodal call + one fallback must fit in 30s.
- **Failover needs both creds present** (Bedrock via role/keys + `GEMINI_API_KEY`).
- **Don't use the full /gsd:* ceremony as the build spine** — plain task-by-task per docs/tasks.md; /gsd:debug, /code-review, /verify as point tools.

## MT10.1 ‚Äî resell sell-to-buyer + real-time inspectable returns + one resell path (June 14)
- **The resell marketplace was list-only ‚Äî no way to accept a buyer.** `MyResells` was a read-only interest feed; `resell.py` listings had no sale concept. Added `status`/`sold_to`/`net_earned` to the listing, `sell_to_interest(lid, interest_id)` (flips listing‚Üísold, accepted buyer, others‚Üí`passed`, net = offer ‚àí delivery_cut) and `decline_interest` (marks one interest `declined`, listing stays active), exposed as `POST /resell/listings/{id}/sell|/decline`. `MyResells` now renders per-buyer Sell/Decline with the take-home, a SOLD summary, and pauses its 3s poll during an action so the optimistic flip isn't clobbered. Verified live: declined Kabir ‚Üí sold to Asha ‚Üí SOLD + ‚Çπ1,683 credited; FlashDeals shows the seller's own listing as "Your listing" and sold ones as "Sold nearby".
- **Buyer returns weren't real-time because `openOps()` overwrote them with a cross-instance refetch.** The buyer `POST /returns` and the Ops `GET /returns` can hit different warm Lambdas (per-instance store), so the new row vanished on the Ops desk. Fix: keep a session-local `localReturns` in App (the optimistic entry, built from the order + catalog item), MERGE it ahead of the server list (dedup by `order_id`, local wins), and never let the fetch clobber it. The row now shows instantly and survives the instance swap. The POST is still fired best-effort so a same-instance refetch also has it.
- **Returned items were display-only ‚Äî you couldn't grade them.** Dynamic return rows rendered as non-clickable `QueuedReturn`. Added `item_id` to the return entry (`returns.py`) so a row that resolves to a catalog item becomes a clickable `GradeableReturn` ("Inspect & grade") that opens the SAME spine as the hero (ItemIntro‚ÜíGrade‚ÜíRoute‚ÜíHealthCard) via `openReturnForGrading` (forces `lane="spine"`, bypassing the LANE map so SL-003 doesn't open Diagnose). Photoless seeded rows stay `QueuedReturn`. **Needed a cached grade for the kurta:** added `cached/SL-003.grade.json` (apparel, grade B, collar defect, "colour not as shown" justification) so the returned-kurta inspection works in BOTH live and CACHED modes (SL-003 has no seed images ‚Üí live grade of no-upload falls through to this cache; before, it 502'd).
- **The user overrode a prior "keep both resell paths separate" decision.** MT10 had kept the notification "Resell now" on the old `/radar`‚Üí`/price-curve` flow while order-history used the new confirm‚Üíphoto‚Üíprice‚Üílist flow. The user wanted ONE path. `openNotif` now calls `startResell` (looking up the matching order in `buyerOrders` for the real paid/date), and `resellOrder` was deleted (orphaned). The Idle Asset Radar beat is still reachable from the Ops inbox SL-002 lane, so that story isn't lost. Lesson: a documented design choice is provisional ‚Äî when the user says unify, unify.

Rather than prop-toggle `WebShell` per screen, the clean split is: landing = its own full-bleed component (no shell), inner pages = the slim `WebShell`. Removed `cartCount`/`onCart` from `WebShell` ‚Äî the buyer cart is still reachable via BuyerStore's in-page Cart tab, so the global cart button was redundant.
- **MT8's derived cascade is only as deep as the LOCKED VRS constants allow ‚Äî and that's correct, not a bug.** `donate` net = 0.15¬∑MRP ‚àí 30 dominates `liquidate` (0.12¬∑MRP ‚àí 20) for every MRP > ~333, and `local_p2p` always beats `warehouse_relist` (higher kept sale, far lower cost). So for SL-001 the cascade is just `local_p2p ‚Üí donate` (2 tiers); no item produces the doc's illustrative 4-tier local‚Üíwarehouse‚Üíliquidate‚Üídonate. Do NOT "fix" this by tuning constants (they're LOCKED, architecture ¬ß4, and the spine reconciliation depends on them). Higher-value/repairable items (SL-006 grade C ‚Üí refurbish‚Üídonate) go deeper. The honest 2-tier shoe waterfall still answers "what happens to items that don't sell locally."
- **Refactor `route_item` ‚Üí `build_paths()` + `grade_for()` to share the path math with cascade, WITHOUT changing spine output.** The cascade re-runs the same argmax at decayed prices, so the path builder had to become a pure, side-effect-free function (no passport writes, takes `resale`/`refurb_resale` as params). Verified the spine is byte-identical after the refactor (local_p2p +‚Çπ83, every eligible path's recovery == sum(breakdown)). The mrp-anchored floors (donate/liquidate/rto) DON'T decay (credit/bulk are on MRP fair value, not time-sensitive resale) ‚Äî that's exactly why the cascade falls THROUGH the decaying resale channels down to the constant donate floor.
- **`/cascade` requires a prior GRADED event (409 otherwise) ‚Äî so it only renders on the spine RouteScreen in the demo.** The RTO lane has no GRADED (sealed units skip grading), and the radar/QUEUED lanes never reach RouteScreen. So App fetches the cascade **non-blocking** after `routeSafe` on the spine lane only and passes `cascade={null}` for RTO. `cascadeSafe` mirrors `routeSafe` (re-grade once on a cold-start 409).
- **The ECR-login 400 is STILL cosmetic (confirmed again this session).** `deploy.ps1` printed the 400 on the login line, then `docker push` succeeded (`latest: digest: sha256‚Ä¶`) and `Done. Deployed`. Check for the digest + "Done", not the login line.

Redefining `.screen-scroll` in `index.css` to a centered reading column (`max-width:44rem; margin-inline:auto`) and deleting `.phone`/`.phone-screen` de-phoned ALL flow/spine screens at once with zero per-screen edits. Fixed it to a light pill. `SLBadge` (mint) and `SourceTag` (mint/grey) already read on light. When restyling a shared chrome component, grep its `right=` slot consumers.
- **Two stacked sticky bars overlap.** `WebShell`'s brand bar is `sticky top-0`; making the per-screen `TopBar` also sticky-top-0 would overlap it. Kept `TopBar` non-sticky (normal flow) ‚Äî the persistent brand bar covers "where am I", the sticky-bottom `FooterAction` covers the CTA. Also moved `RadarToast` + the error note from `absolute` (phone-relative) to `fixed inset-x-0 bottom-0` + `mx-auto max-w-md` so they pin to the viewport, centered, on web.
- **Hybrid photo upload, zero deps:** `lib/image.js` uses a canvas (`drawImage` ‚Üí `toDataURL("image/jpeg", 0.72)`, long edge ‚â§1024px) to downscale a phone photo to ~300 KB base64 before POSTing. Backend `/grade` `current_images` (‚â§3, ‚â§1.5 MB decoded each) decodes+validates BEFORE any LLM call (422 fast-fail), uses them as the CURRENT set against the seeded catalog+day-0 baseline; `_grade_live(item, current_override)`. ## Pre-build gotchas (carried from boilerplate phase ‚Äî verified the hard way)
- **Bedrock throttles new accounts in us-east-1** ‚Üí everything runs in ca-central-1.
- **Nova 2 Lite needs the inference-profile ID** `us.amazon.nova-2-lite-v1:0` (fallback `global.amazon.nova-2-lite-v1:0`), not a bare model ID.
- **Nova 2 is a reasoning model** ‚Üí responses may contain a `reasoningContent` block before the text; extract the block that has `text`, never `content[0]`. Same applies to multimodal/JSON outputs.
- **Lambda container builds need `--provenance=false`** (handled in deploy.ps1); base image already has the RIC; x86_64, 512 MB, 30s timeout.
- **`load_dotenv()` must run before boto3 use locally**, else `NoCredentialsError`.
- **CORS lives in the FastAPI app only**; Function URL CORS stays disabled or it answers preflights itself.
- **Bedrock calls must be time-bounded** (connect 5s / read 15s, 2 retries) so a hung primary fails over to Gemini before the Lambda 30s timeout. Budget: with images, leave headroom ‚Äî one live multimodal call + one fallback must fit in 30s.
- **Failover needs both creds present** (Bedrock via role/keys + `GEMINI_API_KEY`).
- **Don't use the full /gsd:* ceremony as the build spine** ‚Äî plain task-by-task per docs/tasks.md; /gsd:debug, /code-review, /verify as point tools.

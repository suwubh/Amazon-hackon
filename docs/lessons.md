## MT5 — Bulletproof + polish + repo (June 14)
- **Most of MT5 was already done — the win was turning manual checks into committed, regression-guarded artifacts, NOT manufacturing changes.** The FORCE_CACHED kill switch + README/architecture/URLs shipped in the audit-fix pass; the web-console polish in MT13; the loading/error states (try/catch → `setErr` + retry + global `ErrorNote`, skeletons/spinners) were already comprehensive in `App.jsx`. So MT5 added exactly two files — `tests/test_cache_floor.py` + `scripts/failover_drill.py` — with **no app/runtime edit → no redeploy** (the Dockerfile `COPY app` excludes `tests/`+`scripts/`). The lesson: when a "bulletproof" task lands on an already-mature codebase, resist inventing edits; lock the existing robustness behind a test/drill instead.
- **The failover drill must run each stage in a FRESH subprocess — `llm.py` reads `LLM_PRIMARY`/`FALLBACK`/`GEMINI_MODEL`/`BEDROCK_MODEL_ID` at import.** Only `GEMINI_API_KEY` is read per-call. So `failover_drill.py` spawns `python failover_drill.py --worker` 3× with different env. Two gotchas: (1) Stage 1 must NOT set `GEMINI_API_KEY` (let the worker's `load_dotenv()` supply the real one from `.env`) — `load_dotenv` is `override=False`, so an explicitly-set empty/bad key WINS over `.env` (that's exactly why the bad-key stages work, but it silently breaks stage 1 if you pass an empty key). (2) The orchestrator can't read `.env` itself (it never imports `llm`), so don't try to forward the key from the parent — defer to the subprocess.
- **Gemini 2.5 Flash's FREE tier 429-quota-walls under RAPID successive calls — and that's invisible on stage because it auto-fails-over to Nova with the SAME grade.** Running the drill (3 live grades back-to-back) + a few manual grade calls exhausted the per-minute/day quota → every subsequent live grade returned `429 RESOURCE_EXHAUSTED` → transparently served by `live-bedrock` (Nova), still grade D. A SINGLE normal demo grade has headroom (the fresh-verifier subagent saw `Live AI · gemini` working live on Vercel minutes later). So: don't hammer Gemini right before a live demo; and Stage 1 of the drill asserts `live-` (any live provider), not `live-gemini` specifically — whether Gemini or its failover answers, "a live model answered correctly" is the real invariant. Grade D is invariant across gemini/nova/cache, so the hero +₹83/−₹129 never moves.
- **The cache-floor guard is self-maintaining by globbing `seed/cached/*.{grade,seal,diagnose}.json`** — add a new cached item and the coverage extends automatically. It asserts `source=="cached"` + schema-valid in `force_cached=True` mode (which short-circuits before any provider, so it's hermetic/creds-free/offline). Pairs with `test_vrs.py`: that guards the money math, this guards the stage-safety net. Both standalone-runnable (`python tests/test_*.py`) since pytest isn't a project dep.
- **Navigation is 502-proof by construction, worth re-verifying in a bulletproof pass:** the only clickable AI paths in the Inbox are the hero (cached+images), RTO (seal cache), and `item_id`-resolved dynamic returns. The 3 seeded `/returns` rows are `item_id=None` → render as display-only `QueuedReturn` (no click). Fillers SL-005/007/008 have no grade cache but never appear in the inbox (and Shop→PDP is a stateless size-advice read). So no judge click can reach a `CacheMiss` 502.
- **Don't flip repo visibility autonomously.** Making `suwubh/Amazon-hackon` public is in MT5 scope but it's an outward-facing publish — left as a human task (`gh repo edit … --visibility public`). Checked current state read-only (`gh repo view --json visibility` → PRIVATE) to inform the handoff.

## Audit-fix pass — 14 findings (June 14)
- **"Persist everything in DynamoDB" done safely = one best-effort single-table layer that no-ops when the table is unset.** `store.py` reuses the existing `SecondLifePassport` table (PK `item_id`, SK `ts`) with synthetic partition keys (`LISTING`/`RETURNS`/`CART#<persona>`) so it needs no new table — just `dynamodb:GetItem` added to the role (Query/PutItem already granted). The iron rule held by construction: when `DYNAMODB_TABLE_NAME` is unset we never touch boto3 (local dev/tests are byte-identical to before), and when set, every write is fire-and-forget + every read returns `[]`/`None` on failure with short timeouts, so a DynamoDB problem degrades to the old per-instance behaviour instead of blocking the demo. Verify locally by asserting `store.enabled() is False` and that listings/returns counts are unchanged.
- **The spine 409 fix (finding 2) is `latest_event` read-back ONLY — leave `get_events` in-memory.** The cold-start/parallel-instance failure is "grade worked, then route/health-card 409s," i.e. `latest_event("GRADED"/"ROUTED")` missing. Fixing just `latest_event` (fast path = in-memory, miss → query DynamoDB) is surgical. Merging DynamoDB into `get_events` would have changed `/metrics` semantics (it counts off the event log) and broken `POST /metrics/reset` (which only clears in-memory) — so `get_events` stays in-memory. Also: bump the passport `ts` to microsecond resolution or two same-second events overwrite on the (item_id, ts) key.
- **`{/* global error */}` in JSX is an ESLint global-directive landmine.** A JSX comment whose text starts with `global ` is parsed by ESLint as `/* global error */` — declaring `error` as a global, which `no-unused-vars` then flags as "'error' is defined but never used" at a *comment* line. Bewildering until you see it. Fix = reword the comment. (ripgrep showed only the comment; eslint "saw" a variable — that's the tell.)
- **`react-hooks/set-state-in-effect` (new recommended error) fires on a synchronous setState in an effect body, NOT on setState inside an inner async fn.** FlashDeals (inner `async tick()`) passed; MyResells (direct `refresh()`) failed. Fixes: wrap the mount call in an inner `async` tick (MyResells), derive the clamped value in render instead of an effect (ResellPrice), and drop the `loading` boolean in favour of `data === null` so there's no `setLoading(true)` in the effect (YourThings). Real restructures, not disables.
- **Don't store uploaded photos in DynamoDB even when told to "persist everything."** Base64 images (~300 KB each) blow the 400 KB item limit. The grade FACTS persist (small); the uploaded photos for the Health Card live in an in-memory `uploads.py` (per-instance) — fine because the card is built in the same flow on the same warm instance right after grading. A senior call: persist what's small and durable, keep the bulky presentational bytes ephemeral (S3 would be the real home).
- **Resolve PRD `[VERIFY-C]` stats by web-searching the real source, not softening.** NRF/Happy Returns ($890B US returns 2024, 16.9% of sales), Shipway ShipNotes (~26% COD RTO India), Edgistify (₹200–400/return = ₹120–250 ship + ₹80–150 refurb) — all citable, and the ₹200–400 figure happens to bracket the PRD's own ₹220 unit-economics example, so they reinforce each other.

## MT15 — Make the invisible warehouse visible (June 14)
- **`deploy.ps1` can ABORT at the cosmetic ECR-login 400 — run build/push/update directly when it does.** The script's first line is `$ErrorActionPreference = "Stop"`, and in Windows PowerShell 5.1 a native command writing to stderr (the `docker login` 400) is promoted to a *terminating* error → the script dies at the login line and never reaches build/push. But login is **redundant**: `~/.docker/config.json` has `"credHelpers": {"…ecr.ca-central-1…": "ecr-login"}`, so `docker push` mints its own ECR token via `docker-credential-ecr-login` (the MT10 permanent fix). Recovery that works: run the four steps by hand from the `backend/` dir — `docker build --platform linux/amd64 --provenance=false -t <registry>/hackon-backend:latest <backend-path>`, `docker push …`, `aws lambda update-function-code --function-name hackon-backend --image-uri … --region ca-central-1`, `aws lambda wait function-updated …`. Push succeeded (digest `sha256:2e6b6cfd…`), Lambda went Active. The PowerShell tool starts in the **repo root**, not `backend/`, so `docker build .` fails "no Dockerfile" — pass the backend path as the build context. (Confirms the standing lesson: the login 400 is cosmetic; the helper authenticates the push. The only new wrinkle is `ErrorActionPreference=Stop` killing the script before it gets there.)
- **Field-name drift between a fresh backend module and its fresh frontend is the silent killer — the build passes, the UI renders blank.** `your_things`/`green_ledger` were written in the same session as `YourThings.jsx`/`GreenLedger.jsx`; `npm run build` is green regardless of whether `data.total_dormant_value` matches the API's actual key. The cheap guard: curl the deployed endpoint, dump `$r.PSObject.Properties.Name`, and diff against every `data.<key>`/`thing.<key>`/`d.<key>` the components read. Caught nothing this time (all 18 keys aligned) but it's the difference between "verified" and "compiles." Note the list array is `things`, not `items` — easy to mis-probe.
- **Your Things = the pull twin of the Idle-Asset Radar, and it's pure deterministic Python (no LLM/cache → can't fail live).** Same composition pattern as `/second-life` and `/life-stage`: value every owned product through `pricing.resale_value(mrp, category, months_owned, "B", 1.0)` (like-kept grade B), `decay_per_month` = the next-month delta, and reuse the seeded life-stage curve (`seed.lifestage_curve` + `lifestage.DUE_FRACTION`) for `stage_pct`/`due_to_resell`. `total_dormant_value` is the sum of the rows (the iron rule — no hardcoded "your home holds ₹X"). The per-persona Green Ledger scopes `metrics.py`'s CO₂/landfill/diverted math to one persona's `ROUTED` events + a seeded baseline so it reads non-zero pre-demo. Both reuse existing engine pieces — net-new code is thin.

## MT14 — Trust & the closing loop (June 14)
- **A returned item lists on the Flash board reliably ONLY if `/route` does it server-side — the frontend `origin==="ops"` POST was the user-reported bug.** `App.runRoute` fired a *separate* `POST /resell/from-route/{id}` right after routing; on Lambda that second request can hit a different warm instance whose in-memory passport has no `ROUTED` event for the item → `list_from_route` guard fails → silent no-op (the `.catch(()=>{})` hid it). Fix: call `resell.list_from_route(item_id)` **inside `vrs.route_item`**, right after `append_event("ROUTED", …)`, so it's atomic on the same instance. Local import (`from . import resell`) + bare `except: pass` — `resell`→`vrs` has no import cycle, and the board is a side-channel that must never break the spine. Dropped the frontend POST. Lesson: any per-instance write that another endpoint depends on must happen in the SAME request that creates its prerequisite, not a follow-up call.
- **fix-1 (buyer "Order confirmed") needed NO backend change — track the `interest_id`, not the identity.** The accepted-interest status was already in the `/resell/listings` payload (`sell_to_interest` sets the interest `accepted` + listing `sold`/`sold_to`). So the buyer side just had to (a) capture the `interest_id` it created (`addInterest` returns the listing; the new interest is the LAST element), report it to `App`, and (b) poll `/resell/listings` while it has open interests, firing the popup + a sticky `confirmedOrders` notification when its tracked interest flips to `accepted`. No buyer_name/identity plumbing — the interest pool keeps the pretty neighbour names; the buyer matches by id. The poller is `useEffect([myInterests])` that early-returns when empty (no idle polling).
- **Electronics → Amazon Renewed is a NAMED-destination gate, not new economics.** `vrs.build_paths` already attaches `dark_store` (Amazon Now quick-commerce) to an eligible `local_p2p`; MT14 just branches on `category in RENEWED_CATEGORIES={"electronics"}` → attach `renewed_channel` instead (no `dark_store`), and `route_item` adds a top-level `quick_commerce_eligible`. The recovery/breakdown are byte-identical — the VRS reconciliation test still passes for every path. Only the label on `RouteScreen` differs. The trust hook (`usage_cert` battery line) is a seeded field on the electronics catalog items, surfaced by `healthcard.py` (`item.get("usage_cert")`, `None` otherwise) — render-if-present, so non-electronics cards are unaffected.
- **To DEMO electronics→Renewed you need the item to reach RouteScreen → it must be a RETURN with an open window.** Resell doesn't hit `/route` (it goes to ResellPrice/FlashDeals); only the Ops grade→route spine renders RouteScreen. So the electronics order (headphones) was dated recent (`2026-06-10`, ≤ the existing latest order so `_demo_today()` doesn't drift) → return window open → Orders→Return→Ops desk→GradeableReturn→grade→route. SL-006 had no seed images, so a `cached/SL-006.grade.json` (grade B) was required for the live-mode grade to fall through to cache instead of 502'ing.
- **OneDrive transiently reverts files right after a write on this repo path (`C:\Users\subha\OneDrive\…`).** Mid-session, several Edits (vrs.py, items.json, orders.json) + a new file (cached/SL-006.grade.json) showed as REVERTED by a "modified externally" system reminder and a Bash `grep` immediately after — but a settled re-read seconds later showed my edits present (OneDrive conflict resolution flapped, then kept my version). Also hit "File has been modified since read" on a docs edit with identical content (OneDrive touched the mtime). Lesson: after a burst of edits on this path, **re-grep to confirm the settled state before building/deploying**; a single transient "reverted"/"modified" signal is not authoritative — verify, don't re-apply blindly (re-applying could double-edit once the real write lands).

## MT11 — audit backlog (June 14)
- **`/second-life` (buy-side PDP row) is pure composition of existing engine pieces — no new economics.** Price comes from `pricing.resale_value` at the offer's grade + the local demand multiplier (so a judge opening the network tab sees the SAME math the sell side uses); only grade/distance/eta/item_id are seeded facts (`seed/second_life_offers.json`). Returns an **empty `offers` list** for catalog products with no inventory (not 404) so the frontend just hides the row; 404 only for an unknown ASIN. Linking the tap to SL-001's real Health Card would show grade **D** next to an offer that says **C** — a self-inflicted contradiction a judge would catch. A toast (reusing the buyer-flow `RadarToast`) closes the buy-side loop with no passport prereq and no `/metrics` pollution. Reserve a real Health Card link for when each offer maps to its own graded unit.
- **`/metrics` is cumulative per-instance → add a reset, don't fake the number.** The honest fix for the narration drift (MED 5) was two-part: reconcile the demo script to the REAL baseline (`5 items / ₹3,120 / 80% / 9.4 kg`, not the illustrative `8 / ₹4,830 / vs ₹0`), AND add `POST /metrics/reset` → `passport.reset()` which clears the in-memory working set back to the seeded baseline (SOLD/BIRTH/RETURN re-seeded, GRADED/ROUTED dropped). It is **backend-only / not wired to any UI** (a curl-before-the-take presenter tool) so it can't be hit by accident, and it does NOT touch DynamoDB (append-only history). `metrics()` reads only the in-memory passport, so clearing it is sufficient.
- **Kill hardcoded demo dates — derive from the seed.** `orders.py` `TODAY` was a fixed `date(2026,6,13)` that silently closes the return window if the demo slips. `_demo_today()` now derives it from the most-recent seeded order (`latest rahul purchase_date + 2 days`); today's numbers are identical (latest order 06-11 → TODAY 06-13) but it self-updates if the seed dates move. Same pattern beats any "bump the constant near demo day" fix.
- **pytest isn't a project dependency — write the guard test standalone-runnable.** `backend/tests/test_vrs.py` inserts `backend/` on `sys.path` and has a `__main__` block of plain asserts, so it runs via `python tests/test_vrs.py` (what I used) OR `python -m pytest` later. The invariant test (`recovery == sum(breakdown)`) is structural (can only break on a `_path` refactor); the SL-001 **golden** (`==83 / ==-129`) is what actually catches a nudged VRS constant — keep both.

- **Background shells (`run_in_background`) start in a fresh CWD, not the persisted one** — use absolute paths (`cd "C:/.../backend"`) when launching uvicorn/scripts in the background, or you get `cd: backend: No such file or directory`.
- **Windows console is cp1252 — printing the ₹ char (U+20B9) crashes with UnicodeEncodeError.** Set `PYTHONIOENCODING=utf-8` for any python that prints rupee strings (e.g. radar ping_message), or extract numeric fields instead of printing the whole JSON.
- **VRS integrity rule that MT3 depends on:** each path's `recovery` is the exact sum of its `breakdown` values (sale/credit positive, costs negative). The frontend renders the breakdown and the total must reconcile on screen. If you tune a constant, keep this invariant.
- **The api-spec /route example numbers were illustrative and didn't sum** (and assumed grade B). MT2 made them real: the example now reflects the live grade-D output for SL-001 and every figure reconciles. Recovery scales with grade — don't expect the +₹290 "hero" number unless the shoe is graded B (current placeholder cache grades it D → local +₹83).
- **Sealed RTO items skip grading** (architecture §3.2): `/route` normally 409s without a prior grade, but a verified-sealed RTO item routes as factory-new (grade A) with no grade required. So the RTO demo beat is seal-check → route (no scan).
- **In-memory passport is per-Lambda-instance.** `/route` and `/health-card` need the prior `/grade`/`/route` events to be in the SAME warm instance. Sequential demo curls reuse one instance so it works; a cold start (or parallel calls hitting different instances) loses the chain → 409. For a bulletproof stage, set `DYNAMODB_TABLE_NAME` (MT5) or always run the spine in order on one session.

## Pre-build gotchas (carried from boilerplate phase — verified the hard way)
- **Bedrock throttles new accounts in us-east-1** → everything runs in ca-central-1.
- **Nova 2 Lite needs the inference-profile ID** `us.amazon.nova-2-lite-v1:0` (fallback `global.amazon.nova-2-lite-v1:0`), not a bare model ID.
- **Nova 2 is a reasoning model** → responses may contain a `reasoningContent` block before the text; extract the block that has `text`, never `content[0]`. Same applies to multimodal/JSON outputs.
- **Lambda container builds need `--provenance=false`** (handled in deploy.ps1); base image already has the RIC; x86_64, 512 MB, 30s timeout.
- **`load_dotenv()` must run before boto3 use locally**, else `NoCredentialsError`.
- **CORS lives in the FastAPI app only**; Function URL CORS stays disabled or it answers preflights itself.
- **Bedrock calls must be time-bounded** (connect 5s / read 15s, 2 retries) so a hung primary fails over to Gemini before the Lambda 30s timeout. Budget: with images, leave headroom — one live multimodal call + one fallback must fit in 30s.
- **Failover needs both creds present** (Bedrock via role/keys + `GEMINI_API_KEY`).
- **Don't use the full /gsd:* ceremony as the build spine** — plain task-by-task per docs/tasks.md; /gsd:debug, /code-review, /verify as point tools.
